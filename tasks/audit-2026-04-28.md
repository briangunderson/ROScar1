## Critical

### CR-1: `landmark_localizer` yaw correction is meaningless
**File:** [roscar_ws/src/roscar_driver/roscar_driver/landmark_localizer_node.py](roscar_ws/src/roscar_driver/roscar_driver/landmark_localizer_node.py:341-344, 422-427)
**Problem:** The marker pose's orientation is overwritten to identity (lines 341–344) before the TF chain is composed, then `obs_yaw = _yaw_from_matrix(map_marker_mat)` (line 422) reads yaw from that — i.e., from `map→optical` rotation, not the actual marker orientation. So whenever a yaw drift correction fires, the corrected yaw is garbage.
**Fix:** Either preserve `marker.pose.orientation` before chaining, or skip yaw correction entirely (translation-only) by setting `corrected_yaw = base_yaw`.
**Why critical:** Earlier today the user reported "robot pose snapping to wrong locations" during nav with cached markers. This code path could be a contributor.

### CR-2: Worktree ≠ Pi reality (meta)
**Problem:** The Pi has Pi-local launch files (`depth_camera.launch.py`, modified `robot.launch.py`) that weren't in the worktree until I just pulled them. Means: reviewers (humans or agents) reading the worktree see a different launch graph than what runs. Several "missing camera launch" agent findings were artifacts of this.
**Fix:** Commit the pulled files (just done). Going forward, deploy through git rather than direct ssh edits — or at minimum, add a periodic `rsync Pi → worktree → commit` pass.

### CR-3: `slam_nav.launch.py` runs `landmark_localizer` despite documented incompatibility
**File:** [roscar_ws/src/roscar_bringup/launch/slam_nav.launch.py:59-73](roscar_ws/src/roscar_bringup/launch/slam_nav.launch.py:59)
**Problem:** [slam.launch.py:64-68](roscar_ws/src/roscar_bringup/launch/slam.launch.py:64) has a comment block explicitly excluding `landmark_localizer` because *"slam_toolbox's map optimization shifts the map frame, invalidating learned marker positions and causing a feedback loop."* `slam_nav.launch.py` then runs the localizer anyway, with `load_learned: false` — but learned marker positions still get written to disk against a moving map frame.
**Fix:** Remove `landmark_node` from `slam_nav.launch.py`, OR gate it on a "map is stable" flag, OR document why the rule doesn't apply here. The fact that we *just* fixed the cross-session marker contamination suggests this is exactly the loop the comment warned about.

## High

### H-1: AMCL initial pose race
**File:** [roscar_ws/src/roscar_bringup/config/nav2_params.yaml:50](roscar_ws/src/roscar_bringup/config/nav2_params.yaml:50)
**Problem:** `set_initial_pose: true` with default `(0, 0, 0)` AND `launch_manager` publishes `/initialpose` for the slam→nav carryover at t+3/6/10s. AMCL latches the YAML default at activation; the first carryover publish *should* override but races AMCL's subscription readiness.
**Fix:** Set `set_initial_pose: false` and let launch_manager's `/initialpose` carryover win. Add a no-op default sequence at t+12s that also publishes (0,0,0) so manual launches without carryover still work.

### H-2: rosbridge `/map` subscription QoS mismatch
**File:** [roscar_ws/src/roscar_web/web/js/aio-map.js:75-103](roscar_ws/src/roscar_web/web/js/aio-map.js:75)
**Problem:** Slam_toolbox publishes `/map` with `TRANSIENT_LOCAL` durability (latched). Default rosbridge subscription is `RELIABLE` + `VOLATILE`, so a dashboard tab opened *after* the latched message can miss the map entirely until the next periodic republish (5s default). User sees "NO MAP DATA" for the first chunk.
**Fix:** Pass durability hint in the `ROSLIB.Topic` config, e.g. `{ qos: { durability: 'transient_local' } }` (rosbridge_suite >=1.3 supports this).

### H-3: `_save_learned_markers` non-atomic write
**File:** [roscar_ws/src/roscar_driver/roscar_driver/landmark_localizer_node.py:284-295](roscar_ws/src/roscar_driver/roscar_driver/landmark_localizer_node.py:284)
**Problem:** Writes via `open(..., 'w')` and yaml dumps. If the process is killed mid-write (likely during dashboard "Reset Map" or systemd restart), file is left truncated/corrupt and next session crashes in `_load_learned_markers`.
**Fix:** Write to `<file>.tmp` then `os.replace()` to swap atomically.

### H-4: ArUco / YOLO detector callbacks block executor
**Files:**
- [roscar_ws/src/roscar_cv/roscar_cv/aruco_detector_node.py:119-144](roscar_ws/src/roscar_cv/roscar_cv/aruco_detector_node.py:119)
- [roscar_ws/src/roscar_cv/roscar_cv/yolo_detector_node.py:82-108](roscar_ws/src/roscar_cv/roscar_cv/yolo_detector_node.py:82)

**Problem:** Detection runs synchronously inside the image subscriber callback. Subscriptions have shallow queues (depth 1–5). When inference takes >1/15s the executor blocks; old frames either pile up or get silently dropped. Latency grows under load.
**Fix:** Use `MultiThreadedExecutor` + `ReentrantCallbackGroup`, or move detection to a timer that pulls the most-recent cached frame from a single-slot.

### H-5: Touch handlers don't cover nav-goal / init-pose
**File:** [roscar_ws/src/roscar_web/web/js/aio-map.js:629-641](roscar_ws/src/roscar_web/web/js/aio-map.js:629)
**Problem:** After today's `mousedown`/`mouseup`-based refactor of nav-goal click, the touch path only handles pan. Tablet users tapping the map for a nav goal get nothing — `mousedown`/`mouseup` aren't synthesized after `touchstart`/`touchend` since the touch handlers are passive.
**Fix:** Mirror the press-tracking logic into `touchstart`/`touchend` (capture position on touchstart, fire `sendNavGoal`/`sendInitialPose*` on touchend if movement < 12 px).

### H-6: `install_ros2_wsl.sh` writes a CycloneDDS config that contradicts the documented working setup
**File:** [scripts/install_ros2_wsl.sh:71-89](scripts/install_ros2_wsl.sh:71)
**Problem:** Generated config has `<AllowMulticast>true</AllowMulticast>` and no `<Interfaces>` block. CLAUDE.md explicitly documents that WSL2 needs `AllowMulticast=false` AND an explicit `<NetworkInterface address="<LAN IP>" />` block — without it Docker's bridge interfaces win and DDS discovery silently fails.
**Fix:** Update the heredoc to match the documented config; ideally detect the LAN-facing interface IP automatically.

### H-7: ArUco camera intrinsics never refresh
**File:** [roscar_ws/src/roscar_cv/roscar_cv/aruco_detector_node.py:97-103](roscar_ws/src/roscar_cv/roscar_cv/aruco_detector_node.py:97)
**Problem:** `camera_info_cb` early-returns after the first message. If the camera resolution changes (e.g., dashboard switches color profile), pose estimation silently uses the old `K`/`D`.
**Fix:** Cache `(width, height)` and refresh `K`/`D` when those change.

## Medium

### M-1: Driver `_send_motor_cmd` docstring contradicts code
**File:** [roscar_ws/src/roscar_driver/roscar_driver/driver_node.py:259-271](roscar_ws/src/roscar_driver/roscar_driver/driver_node.py:259)
The docstring says "negate wz only," but the code passes `wz` unchanged. After all the chassis-v2 sign work this is a comment that lies to the next reader.
**Fix:** Update docstring to match current behavior.

### M-2: URDF `imu_link` claims aligned with `base_link`, but driver compensates 180° rotation in software
**File:** [roscar_ws/src/roscar_description/urdf/roscar.urdf.xacro:148-152](roscar_ws/src/roscar_description/urdf/roscar.urdf.xacro:148)
URDF says `imu_joint rpy="0 0 0"` but the driver negates ax/ay/az + gx/gy/gz to compensate for the physically-rotated board. Anyone wiring up `imu_filter_madgwick` or another consumer that trusts URDF alignment will get wrong-direction data.
**Fix:** Either set `imu_joint rpy="0 0 ${pi}"` AND drop the driver's manual sign flips (cleaner), or add a prominent comment in the URDF stating "the driver pre-rotates IMU readings — do not transform externally."

### M-3: `_publish_carryover_pose` thread-safety
**File:** [roscar_ws/src/roscar_web/roscar_web/launch_manager_node.py:200](roscar_ws/src/roscar_web/roscar_web/launch_manager_node.py:200)
The bg thread reads `self._current_mode` without a lock. Could publish to `/initialpose` after the user has already toggled away from navigation mode.
**Fix:** Snapshot mode at thread start, or use a `threading.Event` cleared by `_stop_current()`.

### M-4: drawMap allocates new ImageData + offscreen canvas every RAF tick
**File:** [roscar_ws/src/roscar_web/web/js/aio-map.js:280-299](roscar_ws/src/roscar_web/web/js/aio-map.js:280)
In locked-to-robot mode `needsDraw=true` is set on every odom (10Hz) and every TF (10Hz). Re-allocating ImageData and an offscreen canvas 20×/s for an unchanged grid is wasteful.
**Fix:** Cache the offscreen canvas keyed by `(width, height, data-version)`; only repaint it when the `mapSub` callback actually fires.

### M-5: `solvePnP` without `SOLVEPNP_IPPE_SQUARE`
**File:** [roscar_ws/src/roscar_cv/roscar_cv/aruco_detector_node.py:141](roscar_ws/src/roscar_cv/roscar_cv/aruco_detector_node.py:141)
Default `SOLVEPNP_ITERATIVE` can fail or flip orientation for coplanar 4-point inputs (which ArUco markers are by definition).
**Fix:** Pass `flags=cv2.SOLVEPNP_IPPE_SQUARE`.

### M-6: udev rule is overly permissive
**File:** [scripts/udev/99-roscar.rules](scripts/udev/99-roscar.rules)
`MODE="0666"` makes `/dev/roscar_board` world-writable.
**Fix:** `MODE="0660", GROUP="dialout"` and add user to `dialout` group.

### M-7: udev rule matches all CH340 devices
**File:** [scripts/udev/99-roscar.rules:17](scripts/udev/99-roscar.rules:17)
`idVendor==1a86, idProduct==7523` matches every CH340 device on the system. Plug in a second CH340 (Arduino clone, ESP32 dev board) and the symlink races.
**Fix:** Add `ATTRS{serial}=="<board's serial>"` to the rule.

### M-8: setup_rpi.sh re-appends ros2 apt source on rerun
**File:** [scripts/setup_rpi.sh:23-27](scripts/setup_rpi.sh:23)
Unconditional `tee /etc/apt/sources.list.d/ros2.list` — re-running creates duplicate apt repo entries (which apt warns about loudly).
**Fix:** Guard with `[ -f /etc/apt/sources.list.d/ros2.list ]` test.

### M-9: Hardcoded TF frame names in `landmark_localizer`
**File:** [roscar_ws/src/roscar_driver/roscar_driver/landmark_localizer_node.py:351,412,432](roscar_ws/src/roscar_driver/roscar_driver/landmark_localizer_node.py:351)
`'map'`, `'base_footprint'`, `'camera_color_optical_frame'` are all hardcoded.
**Fix:** Declare `map_frame`, `base_frame`, `camera_optical_frame` as parameters; in particular, use `marker.header.frame_id` for the camera frame instead of assuming.

### M-10: TF lookup race in `_set_mode_callback`
**File:** [roscar_ws/src/roscar_web/roscar_web/launch_manager_node.py:160-164](roscar_ws/src/roscar_web/roscar_web/launch_manager_node.py:160)
Uses `rclpy.time.Time()` (latest) with 1.0 s timeout BEFORE killing slam. If slam_toolbox last published map→odom >1 s ago, the lookup falls back silently and the carryover pose may be slightly stale.
**Fix:** Use `Buffer.lookup_transform_async` with a small wait, or cache `map→base_footprint` continuously while slam runs.
