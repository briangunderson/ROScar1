# T2a Proof of Concept — slam_toolbox on WSL2 (2026-05-03)

## Result: ✅ End-to-end works. T2a is no longer hypothetical.

After the network latency pre-flight (PR #36) showed Pi↔WSL2 round-trip
median 4.5 ms / p99 24 ms, this PoC verified the full migration runs:

1. Pi runs `teleop` mode (lidar + EKF + URDF, **no Pi-local slam_toolbox**)
2. WSL2 runs `async_slam_toolbox_node` directly via `ros2 run`
3. Lifecycle transitions (configure → activate) driven via Python service
   calls to `/slam_toolbox/change_state`
4. `/map` (172×127 occupancy grid) published by WSL2, received by Pi
5. `map→odom` TF published by WSL2 at 50 Hz, received by Pi at 50 Hz

## Verification

```text
WSL2 side, after activate:
  /map messages: 2 in 6s (slam_toolbox publishes on map updates, normal)
  map→odom TF samples: 300 in 6s (50 Hz, matches transform_publish_period: 0.02)

Pi side, subscribed to remote topics:
  Pi RECEIVED /map: 172x127 from frame map
  Pi RECEIVED map→odom TFs: 251 in 5s
```

## What this changes

The proposal's T2a was at "pre-flight passed; awaiting go/no-go." This
PoC moves it to "validated on hardware; opt-in plumbing now landed in
this PR."

## Plumbing landed in this PR

### Pi side
- `slam.launch.py` gains a `use_remote_slam` launch arg (default `false`)
- When `true`, the Pi-local `async_slam_toolbox_node` is gated off via
  `UnlessCondition`. The `lifecycle_manager_slam` still runs locally and
  finds the remote `/slam_toolbox` node via DDS — lifecycle services are
  network-transparent, so the manager treats local and remote nodes
  identically.

### WSL2 side
- `roscar_cv/launch/slam_remote.launch.py` — minimal launch that runs
  `async_slam_toolbox_node` with shared config from `roscar_cv/config/`
- `roscar_cv/config/slam_toolbox.yaml` — copy of the Pi's slam_toolbox
  config (anti-drift note in the launch file)
- `roscar_cv/package.xml` — adds `slam_toolbox` as `exec_depend`

### NOT in this PR (deferred to your call)

- ❌ Default flipped to `use_remote_slam:=true`. Default stays `false`
  so existing behavior is unchanged.
- ❌ Dashboard toggle — would require a `SetMode.srv` field. Doable
  but not on the critical path.
- ❌ `roscar-slam.service` systemd unit on WSL2 for always-on. The
  current launch is invoked manually; one user command starts it.
- ❌ `slam_nav.launch.py` opt-in — that path uses `nav2_bringup`'s
  `bringup_launch.py` which spawns slam_toolbox internally; modifying
  that requires a fork or a different approach. Standalone `slam`
  mode opt-in works fine.

## How to flip it on (when you're ready)

```bash
# 1. Deploy slam_toolbox.yaml + slam_remote.launch.py to WSL2:
wsl -d Ubuntu-24.04 -- bash -lc '
  cp /mnt/d/localrepos/ROScar1/roscar_ws/src/roscar_cv/config/slam_toolbox.yaml \
     ~/roscar_ws/src/roscar_cv/config/
  cp /mnt/d/localrepos/ROScar1/roscar_ws/src/roscar_cv/launch/slam_remote.launch.py \
     ~/roscar_ws/src/roscar_cv/launch/
  cd ~/roscar_ws && colcon build --packages-select roscar_cv
'

# 2. Verify WSL2 can launch it:
wsl -d Ubuntu-24.04 -- bash -lc '
  source /opt/ros/jazzy/setup.bash
  source ~/roscar_ws/install/setup.bash
  export RMW_IMPLEMENTATION=rmw_cyclonedds_cpp
  export CYCLONEDDS_URI=file://$HOME/cyclonedds.xml
  ros2 launch roscar_cv slam_remote.launch.py
'
# (Leave this terminal running. /slam_toolbox node will appear, in
# unconfigured state. The Pi will lifecycle-activate it when needed.)

# 3. On the Pi, opt in:
ros2 launch roscar_bringup slam.launch.py use_remote_slam:=true
# OR via the launch_manager:
# (Currently no SetMode field — would need a follow-up PR.)
```

## Expected production benefit

- Pi sheds slam_toolbox's CPU footprint (~7-15 % under load)
- Pi sheds related TF publish overhead (~3-5 %)
- The WSL2 GPU PC has way more headroom — slam_toolbox uses < 1 % of
  one core there
- Net: Pi load avg in slam mode should drop another ~10-15 % on top of
  PR #36's Tier 1 wins

## Risks (and how this PR mitigates each)

| Risk | Mitigation |
|---|---|
| WSL2 not running → no SLAM | Default stays `false`; opt-in is explicit. |
| Network blip → stale TF | p99 round-trip is 24 ms; `transform_tolerance: 1.0` gives 40× headroom (PR #34). |
| Config drift between Pi and WSL2 copies | Anti-drift comment + cp command in deployment doc. |
| slam_nav mode incompatibility | Documented; standalone slam mode works. slam_nav unchanged. |
| Pi's lifecycle_manager bonds to a node it can't ping | `bond_timeout: 0.0` already set in slam.launch.py — exactly the same condition the Pi-local case has. |
